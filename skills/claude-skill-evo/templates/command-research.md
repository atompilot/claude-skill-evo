**ARGUMENTS**: $ARGUMENTS

---

## Step 1：明确调研目标

从用户问题中提取（缺少则推断）：

- **调研对象**：库/框架/服务名称
- **使用场景**：想解决的具体问题
- **对比候选**：用户未提时，主动找 2-3 个主流竞品
- **决策约束**：必须开源、必须支持 {技术栈}、成本限制等

---

## Step 2：并行信息采集

**同时**执行以下搜索，不要串行等待：

**2A — 官方文档 / GitHub README**
- WebFetch 官网文档或 GitHub README
- 提取：定位/用途、安装方式、最后更新时间、Stars

**2B — 包数据**（npm 包适用）
- 月下载量：`WebFetch https://api.npmjs.org/downloads/point/last-month/<包名>`
- bundle 体积：`WebFetch https://bundlephobia.com/api/size?package=<包名>`（提取 gzip、size）

**2C — 技术栈兼容性搜索**（Web 搜索）：
```
"<库名>" {技术栈关键词} compatibility 2025 OR 2026
"<库名>" issues OR problems OR pitfalls
```

**2D — 社区口碑**（Web 搜索）：
```
"<库名>" vs <竞品> reddit OR hackernews 2025 OR 2026
```

**2E — 竞品对比**（若有多个候选）
对每个竞品重复 2A + 2B 关键数据，填入对比表。

---

## Step 3：源码深度分析（按需）

> 需要理解实现细节时使用，文档级调研可跳过。

```bash
mkdir -p reference
git clone --depth 1 {repo_url} reference/{repo_name}
# 特定版本：git clone --depth 1 --branch {tag} {repo_url} reference/{repo_name}
```

**reference/ 规范**：与项目根目录平级，加入 `.gitignore`，分析完可删除。

优先查看：README → 入口文件 → 目录结构 → 依赖文件（package.json / go.mod 等）

| 项目规模 | 建议深度 |
|---------|---------|
| < 1000 行 | 完整阅读 |
| 1000-10000 行 | 核心模块深读 + 周边浏览 |
| > 10000 行 | 目标模块精读 + 架构概览 |
